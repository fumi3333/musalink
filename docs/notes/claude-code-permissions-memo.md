> 後で記事にするときの素材として保存。

最終更新: 2026-05-16

## 1. モードセレクタ（モード）

エディタ右上のモードセレクタから切り替えられる4つのモード。

| モード | キー | 動作 | 使いどころ |
|---|---|---|---|
| **許可を確認** | 1 | ファイル編集・コマンド実行のたびにダイアログが出る | 初めて触る時、未知のリポジトリ |
| **編集を承認** | 2 | ファイル編集は自動承認、コマンドだけ確認 | 普段の作業。バランス型 |
| **プランモード** | 3 | コードは一切変更しない。実装計画だけ立てる | 「まず方針を相談したい」時 |
| **許可をバイパス** | — | 全部自動承認（Claude Code 設定で有効化必要） | フルおまかせ。ただし要注意 |

**所感:** 「編集を承認」がデフォルトで安全寄り。バイパスは個人開発で慣れてきたら検討する程度。

---

## 2. モデルとモードは別物

混同しがちだけど別の概念。

- **モード**: 確認の細かさ（許可の挙動）
- **モデル**: AI 本体（Haiku / Sonnet / Opus）
- **工数（Effort）**: 思考の深さ（低 / 中 / 高 / Max）

普段は **Sonnet 4.6 + 編集を承認 + 工数「中」** で充分。
詰まったときや重要な設計判断のときだけ **Opus 4.7 + 工数「高」** へ。

Opus は Sonnet のおよそ 5 倍コスト感（あくまで体感）。
常用するものではなく、ピンポイント運用が現実的。

## 3. `.claude/settings.local.json` での事前許可

「セッション内は全部許可するから聞かないで」と口頭で言っても、Claude Code のハーネス（実行環境）側のルールには反映されない。
**ハーネスを納得させるには `.claude/settings.local.json` に書く**必要がある。

### 構造

```json
{
  "permissions": {
    "allow": [
      "Bash(pnpm *)",
      "Bash(git status*)",
      "Bash(git push*)"
    ],
    "deny": [
      "Bash(git push --force*)",
      "Bash(firebase deploy*)"
    ]
  }
}
```

### 書き方のコツ

- パターンは glob 風。`Bash(pnpm *)` で `pnpm` から始まるあらゆるコマンドを許可。
- **特定のサブコマンドだけ拒否**したい場合は `deny` に書く（allow より優先される）。
- `Read(**)` でプロジェクト内の全ファイル読み取りを許可（個人開発なら基本これでOK）。

### 実例（Musalink プロジェクトで自分が設定したもの）

**自動許可（ノンストップで進む）**
- ファイル読み取り全般
- `git` 系全般（commit, push, pull, merge, rebase, stash 等）
- `pnpm` / `npm` / `node` / `tsc` / `eslint` / `vitest`
- `firebase` 全般（個別コマンドは deny で絞る）
- `gh` (GitHub CLI)
- 各種 Unix 系（ls, cat, find, grep, jq, diff…）

**自動拒否（必ず確認が必要）**
- `rm -rf *`
- `git push --force` / `git push -f`
- `git reset --hard`
- `git clean -f`
- `git branch -D`
- `firebase deploy*`
- `firebase firestore:delete*`
- `firebase functions:delete*`

## 4. 「ダイアログで止める」 vs 「Claude が日本語で説明する」のトレードオフ

これが今回いちばん腹落ちした話。

### ダイアログの限界

Claude Code は危険な操作の前にダイアログを出してくれる。
ただし**ダイアログには「Yes / No」しかない**。
判断材料は表示されるコマンド文字列だけ。

> 個人開発者として、`firebase deploy --only firestore:rules` が
> 「いま叩いて大丈夫か」を、ダイアログを見た 0.5 秒で判断できるか？
> ……できない。Claude のほうがリポジトリの状態をよく知っている。

つまり、**ダイアログは「ユーザーが詳しい」前提で設計されている**。
個人開発・初心者にとっては、判断を委ねられても困るだけ。

### 本当の安全策はこっち

| 仕組み | 役割 |
|---|---|
| **git のコミット履歴** | ほぼ全てのコード操作は git revert で戻せる |
| **Worktree** | main ブランチに影響しない安全地帯で実験できる |
| **Firestore Rules** | サーバー側で不正書き込みを物理的に防ぐ |
| **テスト環境 / 本番環境分離** | 本物のお金が動く前に必ず止まる |
| **Claude が日本語で説明** | 危険な操作の前に「なぜ危険か」を解説させる |

→ 結論: **ダイアログより、Claude にちゃんと日本語で説明させるほうが重要**。

### 実運用での合意（自分とClaudeの間で決めたルール）

1. 本当にヤバい操作（= 元に戻せない or 他人に影響する）の前は、Claude が必ず日本語で警告する
   - 「これは〇〇という影響があります。実行していいですか？」と書かせる
2. コード変更はガンガン進める（git があるから戻せる）
3. 本番に影響する操作だけは必ず止まる（deny リスト）

## 5. 「取り返しのつかない操作」を見極める

deny リストに入れるべきは、git で戻せないもの・他人に影響するもの。

### 🔴 本当に取り返しがつかない（deny に入れるべき）

- `firebase deploy` 全般 → 本番ユーザーに即影響
- `firebase firestore:delete` → 実データが消える
- `git push --force origin main` → main の履歴破壊、他人にも影響
- `rm -rf` の広範囲 → ファイルが本当に消える
- データベースの直接 DELETE 文
- 外部サービスへのお金が動く API 呼び出し

### 🟡 一見危険だが実は戻せる（許可してOK）

- `git reset --hard`（コミット前ならファイル消えるが、コミット後なら `git reflog` で戻せる）
- `git branch -D`（マージ済みなら問題なし、未マージでも reflog から復旧可能）
- 通常の `git push`（force でなければ既存履歴は壊さない）
- 通常の `rm`（git 管理下なら戻せる）

→ **「コミット済み」「git管理下」「サンドボックス内」のものは、おおむね安全**。

## 6. 個人開発者にとってのベストプラクティス案

記事にするならこの軸でまとめたい：

1. **モードは「編集を承認」で始める** — 慎重すぎず雑すぎず
2. **`.claude/settings.local.json` を整備する** — 「許可をバイパス」より細やかに、かつダイアログから解放される
3. **deny リストは「git で戻せないもの」だけに絞る** — 過剰なdenyは「うっかり防止」より「作業の摩擦」を増やす
4. **Claude には日本語で説明させる** — ダイアログより、コミット前の自然言語の警告のほうが効く
5. **Worktree を活用する** — main を守る最後の砦
6. **CLAUDE.md にプロジェクト固有のルールを書く** — 「これだけは絶対にやらない」を毎回読ませる

## 7. 自分が今回 Musalink でやった具体例

- `CLAUDE.md` 新設（法務レッドライン、環境ルール、IDOR防止ルール、ログ規約）
- `.claude/settings.local.json` を整備（上の §3 の内容）
- deny リストに `firebase deploy*` を入れた（個人開発で本番デプロイは必ず指で止めたい）
- `git push --force*` を deny に入れた（うっかり main を破壊しないため）
